01 — Problem

매 페이지마다 COUNT를 반복한다

일반적인 페이징은 데이터 쿼리와 카운트 쿼리를 동시에 실행합니다. 2페이지 이후에도 totalElements는 바뀌지 않는데, COUNT는 매번 0.3초를 낭비합니다.

↑ 2페이지 이후 COUNT는 동일한 결과(400,254)를 반복 계산 — 순수 낭비

02 — Solution

클라이언트가 totalElements를 돌려보낸다

첫 페이지에서 받은 totalElements를 클라이언트가 보관하고, 다음 요청의 파라미터로 전달하면 서버는 COUNT 쿼리를 건너뜁니다.

1

첫 페이지 요청

totalElements 파라미터 없이 일반 요청

2

COUNT 실행

서버가 카운트 쿼리로 400,254건 계산

3

클라이언트 저장

응답의 totalElements 값을 메모리에 보관

4

다음 페이지 전달

totalElements= 400254 파라미터 추가

5

COUNT 스킵!

04 — Implementation

3개 파일, 10줄 수정

구현 난이도가 매우 낮습니다. DTO에 파라미터 추가, Repository에서 조건 분기, Controller에서 수신 — 끝.

// TransactionSearchRequest.kt // totalElements 파라미터를 추가하여 카운트 쿼리 필요 여부를 판단한다 data class TransactionSearchRequest( val page: Int = 0, val size: Int = 20, val totalElements: Long? = null // 클라이언트가 전달하는 총 건수 ) { /** * 카운트 쿼리 필요 여부 판단 * - totalElements가 null이면 첫 페이지 → 카운트 필요 * - totalElements가 있으면 이후 페이지 → 카운트 불필요 */ fun needsCountQuery(): Boolean = totalElements == null }
// TransactionRepositoryImpl.kt // needsCountQuery()에 따라 카운트 쿼리를 실행하거나 클라이언트 값을 재사용한다 override fun search(request: TransactionSearchRequest): Page<Transaction> { val pageable = PageRequest.of(request.getValidatedPage(), request.getValidatedSize()) // 데이터 조회 (항상 실행) val content = queryFactory .selectFrom(transaction) .where(*whereConditions.toTypedArray()) .orderBy(orderSpecifier) .offset(pageable.offset) .limit(pageable.pageSize.toLong()) .fetch() // 카운트 쿼리 (조건부 실행!) val total = if (request.needsCountQuery()) { // 첫 페이지: 카운트 쿼리 실행 queryFactory .select(transaction.count()) .from(transaction) .where(*whereConditions.toTypedArray()) .fetchOne() ?: 0L } else { // 이후 페이지: 클라이언트가 전달한 값 사용 request.totalElements!! } return PageImpl(content, pageable, total) }
// TransactionController.kt // totalElements를 optional 파라미터로 받아 SearchRequest에 전달한다 @GetMapping fun searchTransactions( // ... 기존 검색 파라미터들 ... @RequestParam(defaultValue = "0") page: Int, @RequestParam(defaultValue = "20") size: Int, @RequestParam(required = false) totalElements: Long? // 추가! ): ResponseEntity<ApiResponse<PageResponse<TransactionListResponse>>> { val request = TransactionSearchRequest( // ... 기존 파라미터들 ... totalElements = totalElements ) // ... }
05 — Trade-offs

알아야 할 것들

이 최적화는 트레이드오프가 있습니다. 적용 전에 반드시 이해하세요.

필수 검색 조건 변경 시 리셋

검색 조건이 바뀌면 totalElements를 반드시 null로 리셋해야 합니다. 다른 조건의 totalElements를 재사용하면 잘못된 페이지 수가 표시됩니다.

주의 데이터 변경 시 불일치

다른 사용자가 데이터를 추가/삭제하면 클라이언트의 totalElements와 실제 건수가 달라집니다. 대부분 허용 가능한 오차입니다.

주의 마지막 페이지 빈 결과

데이터 삭제 후 기존 페이지 수 기준으로 마지막 페이지를 요청하면 빈 결과가 반환될 수 있습니다. 빈 결과 시 리셋하세요.

별도 Deep Pagination은 다른 문제

이 최적화는 COUNT만 스킵합니다. OFFSET이 클 때 데이터 쿼리 자체가 느린 문제는 Cursor 기반 페이징을 검토하세요.

06 — Alternatives

다른 접근법

Keyset

WHERE id < {lastId} 방식으로 OFFSET 없이 페이징합니다. totalElements 자체가 불필요하지만 "5페이지로 점프"가 불가능합니다.

빠름 점프 불가
@Async

데이터를 먼저 반환하고 카운트는 비동기로 처리합니다. 체감 응답 시간은 줄지만 서버 부하는 동일합니다.

UX 개선 부하 동일
EXPLAIN

MySQL EXPLAIN의 rows 값으로 대략적 건수를 추정합니다. 매우 빠르지만 정확도가 낮습니다.

초고속 부정확

간단한 구현, 확실한 효과

3개 파일 수정으로 페이징 성능 46% 개선. 클라이언트가 totalElements를 보관하고 다음 요청에 전달하면 끝. 대부분의 목록 조회 API에 즉시 적용 가능합니다.

구현 난이도: 매우 쉬움 성능 향상: 46% 클라이언트 협업 필요 트레이드오프: 오차 가능
