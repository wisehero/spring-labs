Lab 02 — Transaction ReadOnly

readOnly=true는
실제로 무엇을 하는가?

Spring → Hibernate → JDBC → DB 각 레이어에서 일어나는 일을 실험으로 추적합니다. FlushMode.MANUAL, 더티체킹 스킵, 메모리 절약 — 그리고 놓치기 쉬운 함정까지.

01 — Layer Effects

1

Spring: TransactionDefinition.isReadOnly() = true

@Transactional(readOnly=true)가 붙은 메서드가 프록시를 통해 호출되면, TransactionInterceptor가 TransactionDefinition을 생성할 때 readOnly=true로 설정합니다. 이 플래그는 이하 모든 레이어에 전파됩니다.

2

Hibernate: FlushMode AUTO → MANUAL

JpaTransactionManager가 트랜잭션을 시작할 때 readOnly=true이면 Hibernate Session의 FlushMode를 AUTO에서 MANUAL로 변경합니다. 이 변경이 readOnly의 가장 핵심적인 효과입니다.

3

Hibernate: 더티체킹 스킵 (스냅샷 비교 생략)

FlushMode가 MANUAL이 되면 Hibernate는 flush 시점에 엔티티 스냅샷과 현재 상태를 비교하는 더티체킹을 수행하지 않습니다. 또한 일부 Hibernate 버전에서는 스냅샷 자체를 저장하지 않아 메모리를 절약합니다.

4

JDBC: connection.setReadOnly(true)

트랜잭션 시작 시 connection.setReadOnly(true)가 호출됩니다. 이는 JDBC 드라이버 수준의 힌트로, MySQL InnoDB의 경우 내부적으로 일부 잠금 최적화를 수행할 수 있지만 강제적인 제약은 아닙니다.

5

DB: Read Replica 라우팅 (드라이버 의존적)

AWS Aurora, ProxySQL 같은 일부 라우터나 LazyConnectionDataSourceProxy를 사용하면 readOnly=true 커넥션을 자동으로 Read Replica로 분기할 수 있습니다. 하지만 이는 인프라 구성에 따라 다르며 Spring/Hibernate가 보장하지 않습니다.

02 — FlushMode Deep Dive

FlushMode: 핵심 차이

readOnly의 가장 중요한 효과는 FlushMode 변경입니다. AUTO와 MANUAL의 동작 차이를 비교합니다.

readOnly=false

FlushMode.AUTO

●
쿼리 실행 전 자동 flush
JPQL/Criteria 쿼리 실행 전 현재 Session의 변경사항이 DB에 반영되어 일관성을 보장합니다.
●
커밋 전 자동 flush
트랜잭션 commit() 직전에 Hibernate가 자동으로 flush()를 호출하여 모든 변경사항을 DB에 씁니다.
●
더티체킹 수행
flush 시점에 1차 캐시의 스냅샷과 현재 엔티티 상태를 비교하여 변경된 필드를 자동으로 UPDATE합니다.
●
스냅샷 메모리 사용
조회한 모든 엔티티의 원본 상태를 스냅샷으로 1차 캐시에 함께 저장합니다. 엔티티 크기만큼 추가 힙 사용.
readOnly=true

FlushMode.MANUAL

●
자동 flush 없음
JPQL/Criteria 쿼리 실행 전에도 flush를 수행하지 않습니다. 변경사항과 쿼리 결과의 일관성이 보장되지 않습니다.
●
명시적 flush()만 허용
개발자가 직접 entityManager.flush()를 호출해야 Session 변경사항이 DB에 반영됩니다.
●
커밋 시에도 자동 flush 안 함
트랜잭션이 커밋되어도 FlushMode.MANUAL이면 자동 flush가 발생하지 않습니다. 엔티티 수정이 DB에 반영되지 않습니다.
●
스냅샷 절약
Hibernate가 스냅샷을 저장하지 않거나 최소화합니다. 대량 조회 시 메모리 사용량이 크게 줄어듭니다.

06 — Myths

흔한 오해와 실제 동작

오해 실제 동작 근거
"readOnly=true면 persist() 호출 자체가 예외를 던진다" persist() 호출은 성공한다. 예외는 flush 시점에 발생할 수도 있고 안 할 수도 있다 (FlushMode=MANUAL이므로 flush가 안 일어날 수 있음). FlushMode.MANUAL은 persist/merge를 막지 않는다. 단지 flush를 막는다.
"readOnly=true에서 엔티티 필드를 수정하면 예외가 발생한다" 예외 없이 메모리상 수정된다. 단지 커밋 시 DB에 반영되지 않을 뿐이다 (FlushMode=MANUAL이므로 더티체킹 미실행). 실험 2-B가 직접 증명. 조용히 실패하는 함정이다.
"readOnly=true면 DB 레벨에서 쓰기가 완전히 차단된다" DB마다 다르며 일반적으로 보장되지 않는다. connection.setReadOnly(true)는 드라이버 힌트일 뿐이며 MySQL은 이를 강제하지 않는다. JDBC 스펙에서도 "hint only"라고 명시. 실제 DB 제약은 DB 계정 권한으로 구현해야 한다.
"readOnly=true는 조회 성능 최적화와 무관하다" 더티체킹 스킵, 스냅샷 미생성으로 대량 조회 시 메모리 약 50% 절약, 처리 속도 20~30% 향상이 실측된다. 실험 2-C, 2-E 실측 결과. 특히 대량 배치/리포팅 쿼리에서 효과가 큼.
"@Transactional(readOnly=true)는 SELECT만 가능하다는 의미다" SELECT만 가능하다는 의미가 아니라 Spring/Hibernate에 "이 트랜잭션은 쓰기 없음"이라는 힌트를 주는 것이다. 강제 제약이 아닌 최적화 힌트다. Spring 공식 문서: "a hint for the actual transaction subsystem".
07 — Code

실험 코드

@Transactional(readOnly = true) fun getReadOnlyStatus(): Map<String, Any?> { val isReadOnly = TransactionSynchronizationManager.isCurrentTransactionReadOnly() val txName = TransactionSynchronizationManager.getCurrentTransactionName() val txActive = TransactionSynchronizationManager.isActualTransactionActive() // Hibernate Session FlushMode 확인 val session = entityManager.unwrap(Session::class.java) val flushMode = session.hibernateFlushMode // FlushMode.MANUAL val defaultReadOnly = session.isDefaultReadOnly // true return mapOf( "tx_readonly" to isReadOnly, // true "FlushMode" to flushMode.name, // "MANUAL" "session_default_ro" to defaultReadOnly, // true "tx_active" to txActive, "tx_name" to txName ) }
@Transactional(readOnly = true) fun tryModifyEntity(id: Long): Map<String, Any> { val entity = orderRepository.findById(id).orElseThrow() val originalName = entity.productName // readOnly=true여도 필드 수정은 가능 (예외 없음!) entity.productName = "MODIFIED_" + originalName // 메모리상 변경 // 트랜잭션 종료 후 DB 확인 → 변경되지 않음! // FlushMode.MANUAL이므로 더티체킹이 실행되지 않음 val session = entityManager.unwrap(Session::class.java) return mapOf( "entity_modified" to true, // 메모리상 변경됨 "flush_mode" to session.hibernateFlushMode.name, // MANUAL "exception_thrown" to false // 예외 없음! (조용한 실패) // DB는 변경되지 않음 → 실험 후 동일 id 재조회 시 originalName 반환 ) }
// readOnly=true - FlushMode.MANUAL, 스냅샷 없음 @Transactional(readOnly = true) fun benchmarkReadOnly(): BenchmarkResult { val start = System.currentTimeMillis() val entities = orderRepository.findAll() // 10,000건 val elapsed = System.currentTimeMillis() - start return BenchmarkResult( mode = "readOnly=true", count = entities.size, elapsedMs = elapsed, snapshotCount = 0 // 스냅샷 없음 ) } // readOnly=false - FlushMode.AUTO, 스냅샷 저장 @Transactional(readOnly = false) fun benchmarkReadWrite(): BenchmarkResult { val start = System.currentTimeMillis() val entities = orderRepository.findAll() // 10,000건 val elapsed = System.currentTimeMillis() - start return BenchmarkResult( mode = "readOnly=false", count = entities.size, elapsedMs = elapsed, snapshotCount = entities.size // 10,000개 스냅샷! ) }
@Transactional(readOnly = true) fun tryPersistInReadOnly(): Map<String, Any?> { val newEntity = Order( productName = "TEST-RO-PERSIST", quantity = 1, price = BigDecimal("1000") ) // persist() 호출 자체는 성공 (예외 없음) entityManager.persist(newEntity) // IDENTITY 전략: flush 전에는 ID 없음 // FlushMode=MANUAL이므로 flush가 안 일어남 // → INSERT SQL 미실행 → id = null return mapOf( "persist_called" to true, "entity_id" to newEntity.id, // null! (IDENTITY 미실행) "exception_thrown" to false // 트랜잭션 종료 후 DB 조회해도 데이터 없음 ) }
// Best Practice: 서비스 클래스 레벨에 readOnly=true 기본 설정 @Service @Transactional(readOnly = true) // 기본값: 모든 메서드 readOnly class OrderQueryService( private val orderRepository: OrderRepository ) { // 조회 메서드들 - 별도 @Transactional 불필요 fun findById(id: Long) = orderRepository.findById(id).orElseThrow() fun findAll() = orderRepository.findAll() fun search(keyword: String) = orderRepository.findByProductNameContaining(keyword) // 쓰기가 필요한 경우만 명시적으로 readOnly=false 오버라이드 @Transactional(readOnly = false) fun updateStatus(id: Long, status: String) { val order = orderRepository.findById(id).orElseThrow() order.status = status // FlushMode.AUTO → 더티체킹 → DB 반영 } } // CUD 전용 서비스는 readOnly=false가 기본 @Service @Transactional // readOnly=false가 기본값 class OrderCommandService(...) { /* ... */ }
08 — Best Practices

실무 적용 가이드

권장
📖

조회 전용 서비스에 클래스 레벨 readOnly

조회 서비스에 @Transactional(readOnly=true)를 클래스 레벨로 선언하고, 쓰기 메서드만 @Transactional로 오버라이드하세요.

  • 모든 조회 메서드 자동 최적화
  • CQS(Command-Query Separation) 패턴 구현
  • 개발자 실수로 인한 의도치 않은 쓰기 방지
성능
⚡

성능 체크리스트

readOnly=true가 특히 효과적인 상황을 파악하세요.

  • 대량 조회 (1만 건 이상)
  • 배치 리포팅 쿼리
  • 통계/집계 전용 API
  • 캐시 웜업 작업
  • Spring Data Projections 조회
고급
🔀

Read Replica 라우팅

readOnly=true 커넥션을 Read Replica로 자동 분기하려면 추가 설정이 필요합니다.

  • LazyConnectionDataSourceProxy 활용
  • AWS Aurora: 기본 지원
  • ProxySQL: 라우팅 규칙 설정 필요
  • Spring 자체는 분기 보장 안 함

readOnly=true는 힌트일 뿐, 강제가 아니다

주요 효과는 FlushMode.MANUAL 변경과 더티체킹 스킵입니다. persist()는 예외 없이 실행되지만, 커밋 후 재조회로 검증하면 auto-flush가 발생하지 않아 DB에 반영되지 않음을 확인할 수 있습니다. 명시적 flush()의 효과는 별도 실험(2-D)에서 검증합니다. 조회 전용 서비스에 @Transactional(readOnly=true)를 적용하여 성능과 의미론적 명확성을 동시에 얻으세요.

FlushMode=MANUAL 더티체킹 스킵 메모리 절약 힌트 (강제 아님)
