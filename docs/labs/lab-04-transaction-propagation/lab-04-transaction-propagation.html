Section 01 — Core Concept

두 전파 옵션의 근본적 차이

Spring이 트랜잭션 경계를 어떻게 결정하는가 — 커넥션 공유 vs 커넥션 분리

Propagation.REQUIRED

기존 트랜잭션에 참여

호출 스택에 활성 트랜잭션이 존재하면 그 트랜잭션에 합류합니다. 존재하지 않으면 새 트랜잭션을 시작합니다. Outer와 Inner는 동일한 DB 커넥션과 트랜잭션을 공유합니다.

// 기존 TX 있을 때 Outer TX ──────────────────────────────┐ │ conn-42, tx-A │ └── Inner (REQUIRED) ────────────────┤ │ 같은 conn-42, 같은 tx-A │ └────────────────────────────── COMMIT (tx-A)
// 기존 TX 없을 때 Inner (REQUIRED) ──────────────────────┐ │ 새 conn, 새 tx │ └─────────────────────────────────── COMMIT
Propagation.REQUIRES_NEW

항상 새 독립 트랜잭션

기존 트랜잭션을 suspend(일시 중단)하고, 완전히 새로운 커넥션과 트랜잭션을 시작합니다. Inner가 완료되면 Outer가 resume됩니다. 두 트랜잭션은 완전히 독립적입니다.

// 기존 TX 있을 때 Outer TX ───── [SUSPENDED] ─────────────┐ │ conn-42, tx-A (일시중단) │ └── Inner (REQUIRES_NEW) ────────┐ │ │ 새 conn-99, 새 tx-B │ │ └── COMMIT(tx-B) ──────────┘ │ [RESUMED] tx-A ─────────────────── COMMIT(tx-A)

Section 04 — Deep Dive

실험 4-4: 롤백 트랩 단계별 분석

가장 많은 개발자를 혼란에 빠뜨리는 UnexpectedRollbackException의 발생 메커니즘

1

Outer 메서드 시작 — 새 트랜잭션 생성

Spring TX Interceptor가 새 물리 트랜잭션을 시작. HikariPool에서 conn-42 획득. 행 A 삽입.

// TX Interceptor가 처리 @Transactional // REQUIRED (기본값) fun outerMethod() { repo.save(rowA) // conn-42, tx-A에서 실행 innerMethod() // 내부 호출 }
2

Inner 메서드 — 기존 트랜잭션에 참여

Inner는 REQUIRED이므로 Outer의 tx-A에 합류. 별도 커넥션 없음. 행 B 삽입 후 RuntimeException 발생.

@Transactional // REQUIRED — tx-A에 참여 fun innerMethod() { repo.save(rowB) // 같은 conn-42, tx-A throw RuntimeException("Inner 예외!") }
3

TX Interceptor — rollback-only 마킹!

예외가 Inner TX 경계를 벗어남. TX Interceptor는 현재 물리 트랜잭션(tx-A)에 rollback-only 플래그를 설정. 이 시점에서 tx-A의 운명은 결정됨.

// TransactionAspectSupport 내부 동작 txInfo.getTransactionManager() .rollback(txInfo.getTransactionStatus()) // 논리 TX(Inner)는 참여자이므로: // → 물리 롤백 X, rollback-only 마킹 O status.setRollbackOnly() // ← 핵심!
4

Outer가 예외를 catch — 하지만 이미 늦었다

Outer는 try-catch로 Inner의 예외를 잡아 처리. 개발자는 정상 흐름이라고 생각하지만, tx-A에는 이미 rollback-only 플래그가 설정되어 있음.

try { innerMethod() } catch (e: RuntimeException) { log.warn("예외 처리 완료. 계속 진행...") // ← 개발자는 괜찮다고 생각! // ← 하지만 tx-A는 이미 rollback-only! }
5

Outer 커밋 시도 → UnexpectedRollbackException!

Outer 메서드 종료 시 TX Interceptor가 commit() 호출. TransactionManager가 rollback-only 플래그를 발견하고 실제로는 롤백 수행 후 예외를 던짐. 행 A, B 모두 롤백됨.

// AbstractPlatformTransactionManager if (status.isRollbackOnly()) { processRollback(status, false) // unexpected=true → 예외 발생! throw UnexpectedRollbackException( "Transaction silently rolled back") } // 결과: 행 A, 행 B 모두 DB에서 사라짐
6

최종 결과: 전체 롤백

Outer가 try-catch로 예외를 잡았음에도 불구하고, 행 A와 행 B 모두 데이터베이스에서 롤백됨. 애플리케이션은 저장 성공이라고 생각하지만 DB에는 아무것도 없음.

Section 05 — Comparison

REQUIRED vs REQUIRES_NEW 비교표

두 전파 옵션의 핵심 차이를 모든 시나리오에 걸쳐 정리

비교 항목 REQUIRED REQUIRES_NEW
기존 TX 존재 시 기존 TX에 참여 (같은 커넥션) 기존 TX suspend → 새 TX 시작
기존 TX 없을 시 새 TX 생성 새 TX 생성
DB 커넥션 1개 (공유) 2개 (각각 독립)
Inner 예외 → Outer catch 시 rollback-only 마킹 → Outer 커밋 시 UnexpectedRollbackException Inner만 롤백, Outer 정상 커밋 가능
Outer 예외 → Inner 데이터 함께 롤백 (같은 TX) Inner는 이미 커밋됨 (독립 TX) — 데이터 불일치 가능
커넥션 풀 영향 영향 없음 (1커넥션) 커넥션 2개 동시 사용 → 중첩 루프 시 풀 고갈 위험
트랜잭션 격리 완전 공유 — 운명 공동체 완전 독립 — 각자 commit/rollback
대표 사용 사례 일반 비즈니스 로직 (기본값) 감사 로그, 알림 발송, 이력 저장
Section 06 — Best Practices

실무 적용 가이드

트랜잭션 전파 옵션을 안전하게 사용하기 위한 핵심 원칙

🛡️

REQUIRED를 기본값으로

명시적 이유 없이 REQUIRES_NEW를 사용하지 마세요. REQUIRED는 단일 원자적 작업을 보장하며 커넥션 효율이 높습니다.

⚠️

try-catch 후 커밋 가정 금지

REQUIRED Inner에서 예외를 catch했더라도 트랜잭션은 이미 rollback-only 상태일 수 있습니다. 예외 전파 경로를 항상 추적하세요.

🔌

REQUIRES_NEW와 커넥션 풀

REQUIRES_NEW는 동시에 커넥션을 2개 사용합니다. 중첩이 깊거나 루프 안에서 사용하면 커넥션 풀이 고갈될 수 있습니다.

📋

REQUIRES_NEW의 올바른 사용 사례

메인 로직 실패와 무관하게 반드시 저장되어야 하는 감사 로그, 실패 이력, 알림 발송 등에 적합합니다.

🔍

데이터 일관성 검토

REQUIRES_NEW 사용 시 Outer 롤백 후 Inner 데이터가 남을 수 있습니다. 이 시나리오에서 데이터 일관성이 허용되는지 반드시 검토하세요.

🧪

p6spy로 커넥션 확인

REQUIRES_NEW 사용 시 p6spy 로그에서 두 개의 커넥션 ID를 확인하세요. 예상과 다른 커넥션 패턴은 전파 설정 오류의 신호입니다.

REQUIRED가 기본값인 이유를 이해하고,
REQUIRES_NEW는 신중하게 사용하라
REQUIRED 기본값 rollback-only 마킹 REQUIRES_NEW 독립 커밋 UnexpectedRollbackException 커넥션 풀 고갈 방지 감사 로그 패턴 9가지 실험 검증
