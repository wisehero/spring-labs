Section 01

N+1 발생 메커니즘

LazyProxy는 처음에는 SQL을 실행하지 않는다. 컬렉션에 접근하는 순간 N개의 SELECT가 추가로 실행된다.

01
findAll() 호출 → Team SELECT
SELECT * FROM team -- 1개 쿼리
5개의 Team 엔티티가 로딩된다. members 필드는 LazyProxy로 초기화됨. 아직 SQL 없음.
02
team[0].members.size() 접근 → 추가 SELECT
SELECT * FROM member WHERE team_id = 1 -- 2번째 쿼리
LazyProxy가 초기화되며 PersistenceContext에 Members를 로딩한다.
03
team[1], team[2], ... 각각 추가 SELECT
SELECT * FROM member WHERE team_id = 2 SELECT * FROM member WHERE team_id = 3 SELECT * FROM member WHERE team_id = 4 SELECT * FROM member WHERE team_id = 5
Team 수(N=5)만큼 SELECT가 반복 실행된다.
04
총 쿼리: 1 + N = 1 + 5 = 6 SQL
Team이 100개라면 101 SQL. 1000개라면 1001 SQL. N이 커질수록 선형으로 증가한다.
⚠️
EAGER로 바꾸면 해결되는가?
아닙니다. JPQL 사용 시 EAGER여도 추가 SELECT로 로딩합니다. Spring Data JPA의 findAll()은 내부적으로 JPQL을 사용하기 때문에, FetchType.EAGER 설정이 N+1을 막지 못합니다. Hibernate는 JPQL 결과를 먼저 받은 뒤, EAGER 연관관계를 별도 SELECT로 즉시 로딩합니다.
Section 02

해결 방법 비교

각 방법의 SQL 수, JOIN 타입, 특징을 한눈에 비교한다.

방법 SQL 수 JOIN 타입 특징
LAZY (기본) 1 + N — N+1 발생, 컬렉션 접근 시 추가 SELECT
JOIN FETCH 1 INNER JOIN DISTINCT 필요, members 없는 팀 제외됨
@EntityGraph 1 LEFT OUTER JOIN 어노테이션만으로 설정, members 없는 팀도 포함
DTO Projection 1 JOIN 엔티티 미사용, PersistenceContext 오버헤드 없음
@BatchSize 1 + ⌈N/size⌉ — IN절 배치, 완전한 해결책은 아님
Section 03

6가지 실험

각 실험은 N+1의 발생 패턴 또는 해결 방법 하나를 집중적으로 검증한다.

실험 7-1
@OneToMany N+1 발생
N+1

Team → Members 컬렉션에 LazyProxy가 적용된 상태에서 findAll() 후 각 팀의 members에 접근한다.

⚡ 결과: 1 + 5 = 6 SQL 실행
실험 7-2
JOIN FETCH 해결
INNER JOIN

JPQL에 JOIN FETCH를 명시하여 Team과 Members를 한 번의 쿼리로 로딩한다. DISTINCT로 중복 제거 필수.

✅ 결과: 1 SQL, DISTINCT 적용
실험 7-3
@EntityGraph 해결
LEFT OUTER JOIN

Repository 메서드에 @EntityGraph를 붙여 자동 LEFT OUTER JOIN. JPQL 수정 없이 어노테이션만으로 적용.

✅ 결과: 1 SQL, 어노테이션만으로 완료
실험 7-4
@ManyToOne N+1 (캐시)
1차 캐시

Member → Team 방향의 N+1. 동일 트랜잭션 내에서는 PersistenceContext 1차 캐시에 의해 Team이 재사용된다.

⚡ 결과: 중복 Team은 캐시 재사용
실험 7-5
DTO Projection
PC 없음

JPQL에서 new DTO(...)로 직접 값만 조회. PersistenceContext에 엔티티를 올리지 않아 메모리 오버헤드가 없다.

✅ 결과: 1 SQL, PC 오버헤드 없음
실험 7-6
MultipleBagFetchException
예외 발생

두 개의 List 타입 컬렉션을 동시에 JOIN FETCH 시 Hibernate가 MultipleBagFetchException을 던진다. Set 또는 순차 fetch로 해결.

⚠️ 결과: 예외 → Set 변환 또는 순차 fetch

Section 05

JOIN FETCH vs @EntityGraph

같은 결과처럼 보이지만, JOIN 타입과 엔티티 포함 여부에서 중요한 차이가 있다.

JOIN FETCH

JPQL에 직접 명시하는 방식

JOIN 타입 INNER JOIN
적용 방식 JPQL 직접 수정
members 없는 팀 결과에서 제외됨
중복 처리 DISTINCT 명시 필요
커스텀 가능 WHERE, ORDER BY 자유롭게

@EntityGraph

어노테이션으로 fetch plan 지정

JOIN 타입 LEFT OUTER JOIN
적용 방식 메서드 어노테이션만
members 없는 팀 결과에 포함됨
중복 처리 Spring Data가 처리
커스텀 가능 제한적 (Spec 조합 가능)
Section 06

코드 예제

각 해결 방법의 실제 구현 코드. Kotlin + Spring Data JPA 기준.

@Entity
class Team(
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long = 0,

    val name: String,

    // LAZY 로딩 (기본값, 명시 권장)
    @OneToMany(mappedBy = "team", fetch = FetchType.LAZY)
    val members: MutableList<Member> = mutableListOf()
)

@Entity
class Member(
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long = 0,

    val name: String,
    val role: String,

    // ManyToOne도 기본은 EAGER이므로 LAZY로 명시
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "team_id")
    val team: Team
)
// N+1이 발생하는 코드
@Service
class NplusOneExperimentService(
    private val teamRepository: TeamRepository
) {
    @Transactional(readOnly = true)
    fun experimentNplusOne(): List<TeamResult> {
        // SELECT * FROM team → 1 쿼리
        val teams = teamRepository.findAll()

        // 각 팀의 members에 접근할 때마다 추가 SELECT → N 쿼리!
        return teams.map { team ->
            TeamResult(
                teamName = team.name,
                memberCount = team.members.size,  // HERE: LazyProxy 초기화
                members = team.members.map { it.name }
            )
        }
        // 총 SQL: 1 + N (teams.size)
    }
}
// JOIN FETCH로 한 번에 해결
@Repository
interface TeamRepository : JpaRepository<Team, Long> {

    // DISTINCT: JOIN 결과 중복 제거 (Hibernate 6+에서는 자동)
    @Query("""
        SELECT DISTINCT t FROM Team t
        JOIN FETCH t.members
    """)
    fun findAllWithMembersJoinFetch(): List<Team>
}

// 서비스에서 사용
@Transactional(readOnly = true)
fun experimentJoinFetch(): List<TeamResult> {
    // SELECT DISTINCT t.*, m.* FROM team t INNER JOIN member m ON t.id=m.team_id
    val teams = teamRepository.findAllWithMembersJoinFetch()
    // 총 SQL: 1 (members는 이미 로딩됨, LazyProxy 초기화 없음)
    return teams.map { team ->
        TeamResult(team.name, team.members.size, team.members.map { it.name })
    }
}
// @EntityGraph: 어노테이션만으로 LEFT OUTER JOIN 적용
@Repository
interface TeamRepository : JpaRepository<Team, Long> {

    // attributePaths에 지정한 연관관계를 eager로 fetch
    @EntityGraph(attributePaths = ["members"])
    override fun findAll(): List<Team>

    // 특정 조건과 함께 사용 가능
    @EntityGraph(attributePaths = ["members"])
    fun findByName(name: String): Optional<Team>
}

// JOIN FETCH와의 차이: LEFT OUTER JOIN → members 없는 팀도 포함
// 생성된 SQL:
// SELECT t.*, m.* FROM team t LEFT OUTER JOIN member m ON t.id = m.team_id
// DTO Projection: PersistenceContext 없이 값만 조회
data class TeamMemberDto(
    val teamName: String,
    val memberName: String,
    val memberRole: String
)

@Repository
interface TeamRepository : JpaRepository<Team, Long> {

    @Query("""
        SELECT new com.example.dto.TeamMemberDto(
            t.name, m.name, m.role
        )
        FROM Team t JOIN t.members m
    """)
    fun findAllTeamMembersAsDto(): List<TeamMemberDto>
}

// 장점: 엔티티를 PersistenceContext에 올리지 않음
// → 변경 감지(dirty checking) 오버헤드 없음
// → 읽기 전용 조회에 최적화
// 단점: 엔티티 조작 불가 (업데이트, 연관관계 변경 등)
// @BatchSize: IN절 배치로 N+1 완화 (완전한 해결책은 아님)
@Entity
class Team(
    ...
    @BatchSize(size = 100)  // 최대 100개씩 IN절로 묶음
    @OneToMany(mappedBy = "team", fetch = FetchType.LAZY)
    val members: MutableList<Member> = mutableListOf()
)

// 또는 application.properties 전역 설정:
// spring.jpa.properties.hibernate.default_batch_fetch_size=100

// 동작: 100개 Team 조회 시 members를 100개씩 IN절로 배치 조회
// SELECT * FROM member WHERE team_id IN (1,2,...,100)
// N+1 → 1 + ⌈N/100⌉ 으로 개선
// 단, 완전한 1 SQL은 아님. JOIN FETCH가 더 효율적
Section 07

Best Practices

실무에서 검증된 N+1 대응 전략.

⚡

기본 전략: LAZY + 필요 시 JOIN FETCH

모든 연관관계는 FetchType.LAZY로 설정한다. 컬렉션이 필요한 서비스에서만 JOIN FETCH 쿼리를 별도로 작성해 명시적으로 로딩한다. 예측 가능한 SQL 실행을 보장한다.

📦

읽기 전용 API는 DTO Projection 사용

데이터를 화면에 뿌리기만 하는 조회 API는 엔티티 대신 DTO로 직접 조회한다. PersistenceContext 오버헤드를 줄이고 필요한 컬럼만 SELECT해 네트워크 비용도 절약한다.

🔗

복수 컬렉션은 Set 또는 순차 fetch

두 개 이상의 List 컬렉션을 동시에 JOIN FETCH하면 MultipleBagFetchException이 발생한다. 컬렉션 타입을 Set으로 변경하거나, 첫 번째 컬렉션을 JOIN FETCH한 뒤 두 번째는 별도 쿼리로 로딩한다.

🚫

EAGER 절대 금지

FetchType.EAGER는 JPQL 사용 시에도 N+1을 막지 못한다. 연관 데이터가 필요 없는 상황에서도 항상 JOIN하므로 불필요한 데이터 로딩이 발생한다. 코드베이스에서 EAGER를 완전히 제거한다.

"FetchType.LAZY + 필요 시 JOIN FETCH가 정답.
EAGER는 절대 금지."

N+1은 지식의 문제다. 발생 원인을 이해하면 항상 막을 수 있다.

LAZY 기본 JOIN FETCH 1순위 DTO 읽기전용 EAGER 금지 MultipleBag → Set @BatchSize 보조
