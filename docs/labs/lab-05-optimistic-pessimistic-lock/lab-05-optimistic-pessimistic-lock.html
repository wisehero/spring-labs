01 · 핵심 개념

낙관적 락 vs 비관적 락

Optimistic Lock

낙관적 락 — @Version

→ SELECT id, stock, version=3 FROM product WHERE id=1
→ 비즈니스 로직 처리 (재고 -1)
→ UPDATE product SET stock=9, version=4
   WHERE id=1 AND version=3
→ 1 row affected? 성공 ✓
→ 0 row affected? OptimisticLockException ✗
→ 충돌 발생 시 → 재시도 (Retry)
Pessimistic Lock

비관적 락 — FOR UPDATE

→ TX-A: SELECT ... FROM product WHERE id=1 FOR UPDATE
→ TX-A: Row Lock 획득 — stock=10 읽기
→ TX-B: SELECT ... FOR UPDATE
   → WAITING... (블로킹)
→ TX-A: UPDATE stock=9 → COMMIT (Lock 해제)
→ TX-B: 깨어남 — 최신 stock=9 읽기
→ TX-B: UPDATE stock=8 → COMMIT ✓

05 · 전략 가이드

전략 선택 가이드

구분 낙관적 락 (Optimistic) 비관적 락 (Pessimistic)
구현 방식 @Version 필드 + WHERE version=? SELECT ... FOR UPDATE (DB 레벨 락)
락 시점 COMMIT 시점에 충돌 감지 SELECT 시점에 즉시 락 획득
충돌 처리 OptimisticLockException → Retry 다른 TX는 대기 (Blocking)
낮은 경합 환경 우수 락 오버헤드 없음 보통 불필요한 락 비용
높은 경합 환경 불리 재시도 폭증 우수 순서 보장, 정확성
데드락 위험 없음 존재 획득 순서 관리 필요
DB 부하 낮음 (충돌 없을 때) 중간 락 관리 오버헤드
분산 환경 적합 DB 락 불필요 주의 DB 연결 유지 필요
주요 적용 사례 상품 정보 수정, 게시글 편집, 설정 변경 재고 차감, 포인트 사용, 좌석 예약

낙관적 락 추천 시나리오

  • 동일 데이터 수정 충돌이 드문 경우 (충돌률 < 10%)
  • 사용자가 오랜 시간 편집하는 폼 기반 UI
  • 읽기가 많고 쓰기가 적은 CRUD 서비스
  • 마이크로서비스 간 분산 트랜잭션 불가 환경
  • 재시도 로직 구현이 가능한 경우

비관적 락 추천 시나리오

  • 플래시 세일, 한정 수량 재고처럼 경합이 극심한 경우
  • 재시도 비용이 높거나 멱등성 보장이 어려운 경우
  • 정확한 순서 보장이 필요한 포인트/잔액 처리
  • 트랜잭션이 짧고 빠르게 완료되는 경우
  • 데드락 방지를 위해 획득 순서를 전역 통제 가능한 경우

"경합 빈도에 따라 최적 전략이 달라진다"

낙관적 락은 충돌이 드물 때, 비관적 락은 경합이 심할 때 빛을 발한다.
두 전략을 혼합해 계층별로 적용하는 것이 실무의 정답이다.
