Section 01
Caffeine Cache란?

Java 진영에서 가장 빠른 로컬 캐시 라이브러리로, Window TinyLFU 퇴거 알고리즘으로 높은 히트율을 보장한다.

왜 Caffeine인가?

  • Guava Cache 후속
    Google Guava의 캐시 모듈을 완전히 재설계. 동일 API, 더 나은 성능.
  • Near-optimal Hit Rate
    Window TinyLFU 알고리즘으로 LRU, LFU보다 높은 히트율 달성.
  • O(1) 연산
    get, put, invalidate 모두 상수 시간. ConcurrentHashMap 기반.
  • Spring Boot 기본 제공
    spring-boot-starter-cache + caffeine 의존성만 추가하면 @Cacheable 즉시 사용.

로컬 캐시 vs 분산 캐시

Caffeine (로컬) Redis (분산)
저장 위치 JVM 힙 메모리 별도 서버
지연 시간 ~ns (메모리 직접 접근) ~ms (네트워크 왕복)
서버 간 공유 불가 (프로세스 격리) 가능
적합한 경우 단일 서버, 읽기 집중 다중 서버, 일관성 필요

Section 04
Window TinyLFU 알고리즘

Caffeine이 LRU보다 높은 히트율을 달성하는 이유 — 빈도(frequency) + 최근성(recency)을 동시에 고려.

Window TinyLFU란?

전통적인 캐시 퇴거 알고리즘인 LRU(Least Recently Used)는 최근성만 고려하고, LFU(Least Frequently Used)는 빈도만 고려한다. 두 방식 모두 한계가 있다. LRU는 한 번 스캔된 대량 데이터가 자주 쓰는 데이터를 밀어내고, LFU는 과거에 자주 쓰였지만 더 이상 쓰이지 않는 데이터가 자리를 차지한다.

Window TinyLFU는 이 둘을 결합한다. 새 항목은 먼저 작은 Admission Window(~1%)에 들어가고, 여기서 살아남으면 Probationary 영역으로 이동한다. 이때 TinyLFU 필터가 새 항목과 퇴거 후보의 접근 빈도를 비교하여, 빈도가 낮은 쪽을 퇴거한다.

빈도 추적에는 4-bit CountMinSketch를 사용한다. 항목당 4비트만 사용하므로 메모리 오버헤드가 극히 작고, O(1) 시간에 빈도를 조회할 수 있다. 주기적으로 카운터를 절반으로 감소시켜(aging) 오래된 빈도가 과대 평가되는 것을 방지한다.

해결 전략

🔒
분산 락: @DistributedLock / Redis SETNX로 단일 쓰레드만 DB 조회, 나머지는 대기 후 캐시 재사용
⏱️
TTL 지터(Jitter): expireAfterWrite에 랜덤 오프셋 추가 → 만료 시점 분산
🔄
Refresh-Ahead: refreshAfterWrite로 만료 전 비동기 갱신 → 항상 유효한 값 유지
📋
Probabilistic Early Expiration: TTL 만료 근접 시 확률적으로 일부 쓰레드만 갱신 담당
Section 07
Best Practices

프로덕션에서 Caffeine Cache를 안전하게 활용하기 위한 핵심 가이드라인.

🎯
캐시 대상 선정
  • 읽기 빈도 높고 변경 드문 데이터
  • 계산 비용이 큰 집계 쿼리
  • 외부 API 응답 (Rate Limit 고려)
  • 사용자별 개인화 데이터는 제외 권장
  • null 반환 가능성 → unless="#result==null"
⏰
TTL 전략
  • 정적 데이터: 1시간~24시간
  • 준정적 데이터: 5분~30분
  • 실시간성 필요: 30초~2분
  • Jitter 10~20% 추가로 Stampede 방지
  • refreshAfterWrite로 Stale-While-Revalidate 구현
🛡️
Stampede 방지
  • Critical 캐시에 분산 락 적용
  • TTL Jitter로 만료 시점 분산
  • Connection Pool 크기 × 2 이상 여유 확보
  • Circuit Breaker와 조합 (Resilience4j)
  • Soft TTL / Hard TTL 이중 전략
⚠️
Self-invocation 주의
  • 같은 클래스 내부 호출은 프록시 미통과
  • @Cacheable이 무시됨
  • Self-injection 패턴으로 해결
  • 또는 별도 Bean으로 분리
  • @Lazy @Autowired lateinit var self
핵심 원칙
Cache HIT 시 메서드 본문이 실행되지 않는다는 점이 핵심 —
DB가 아예 존재하지 않는 것처럼 동작한다.
