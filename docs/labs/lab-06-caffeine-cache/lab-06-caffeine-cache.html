HIT / MISS 판단
⚡
CACHE HIT — 메서드 미실행
1. 프록시가 메서드 호출을 인터셉트
2. cache.get(key) → 값 존재
3. 메서드 본문을 실행하지 않음
4. 캐시 값을 즉시 반환 (DB 쿼리 없음)
🐢
CACHE MISS — DB 조회 후 저장
1. 프록시가 메서드 호출을 인터셉트
2. cache.get(key) → null
3. 메서드 본문 실행 (DB SELECT)
4. 결과를 cache.put(key, value)로 저장

Section 03
Window TinyLFU 알고리즘

Caffeine이 LRU보다 높은 히트율을 달성하는 이유 — 빈도(frequency) + 최근성(recency)을 동시에 고려.

80%

승격 / 강등 규칙

1
새 항목 진입: Window 영역에 배치. LRU 방식으로 관리됨.
2
Window 만료: TinyLFU frequency filter 통과 시 Probationary로 이동.
3
재접근 시 승격: Probationary에서 hit 발생하면 Protected로 승격.
4
Protected 만료: 공간 부족 시 Probationary로 강등 후 eviction 경쟁.

Section 05 — 경고
Cache Stampede

TTL이 만료되는 순간, 모든 대기 중인 스레드가 동시에 DB를 공격한다.

TTL 만료
다수 쓰레드
동시 대기 중
→→
Cache MISS × N
다수 쓰레드가
캐시에서 null 반환
받음
→→
DB 동시 폭격
SELECT × N
Connection Pool
고갈 → Timeout

해결 전략

🔒
분산 락: @DistributedLock / Redis SETNX로 단일 쓰레드만 DB 조회, 나머지는 대기 후 캐시 재사용
⏱️
TTL 지터(Jitter): expireAfterWrite에 랜덤 오프셋 추가 → 만료 시점 분산
🔄
Refresh-Ahead: refreshAfterWrite로 만료 전 비동기 갱신 → 항상 유효한 값 유지
📋
Probabilistic Early Expiration: TTL 만료 근접 시 확률적으로 일부 쓰레드만 갱신 담당
Section 06
Best Practices

프로덕션에서 Caffeine Cache를 안전하게 활용하기 위한 핵심 가이드라인.

🎯
캐시 대상 선정
  • 읽기 빈도 높고 변경 드문 데이터
  • 계산 비용이 큰 집계 쿼리
  • 외부 API 응답 (Rate Limit 고려)
  • 사용자별 개인화 데이터는 제외 권장
  • null 반환 가능성 → unless="#result==null"
⏰
TTL 전략
  • 정적 데이터: 1시간~24시간
  • 준정적 데이터: 5분~30분
  • 실시간성 필요: 30초~2분
  • Jitter 10~20% 추가로 Stampede 방지
  • refreshAfterWrite로 Stale-While-Revalidate 구현
🛡️
Stampede 방지
  • Critical 캐시에 분산 락 적용
  • TTL Jitter로 만료 시점 분산
  • Connection Pool 크기 × 2 이상 여유 확보
  • Circuit Breaker와 조합 (Resilience4j)
  • Soft TTL / Hard TTL 이중 전략
⚠️
Self-invocation 주의
  • 같은 클래스 내부 호출은 프록시 미통과
  • @Cacheable이 무시됨
  • Self-injection 패턴으로 해결
  • 또는 별도 Bean으로 분리
  • @Lazy @Autowired lateinit var self
핵심 원칙
Cache HIT 시 메서드 본문이 실행되지 않는다는 점이 핵심 —
DB가 아예 존재하지 않는 것처럼 동작한다.
