Lab 01 — Spring AOP Proxy

this.method()는 왜
@Transactional을
무시하는가?

같은 클래스 내부에서 호출하면 CGLIB 프록시를 우회합니다. REQUIRES_NEW, readOnly, timeout — 모든 속성이 조용히 무시됩니다. 런타임 예외 없이.

01 — Core Mechanism

AOP 프록시 동작 원리

Spring은 @Transactional이 붙은 Bean을 프록시로 감쌉니다. 외부 호출은 프록시를 통과하지만, 내부 호출은 우회합니다.

02 — Why It Happens

왜 this.method()는 프록시를 우회하는가

6단계로 추적하면 문제의 원인이 명확해집니다.

1

Controller가 프록시를 호출

proxy.experimentSelfInvocation() — Controller가 주입받은 것은 프록시 인스턴스

2

CGLIB 프록시가 호출을 가로챔

프록시의 메서드 오버라이드가 실행됨. 인터셉터 체인으로 진입

3

TransactionInterceptor가 트랜잭션 시작

JpaTransactionManager.doBegin() → tx name 설정 → ThreadLocal에 메타데이터 저장

4

실제 target 객체의 메서드 호출

이 시점에서 this = target 객체 (프록시가 아님!). 프록시는 target.method()를 호출하기 때문

5

this.innerMethod() → 프록시 우회!

this는 raw target이므로 TransactionInterceptor가 실행되지 않음. @Transactional(REQUIRES_NEW) 완전 무시

6

innerMethod()는 일반 메서드 호출처럼 실행

새 트랜잭션도, 읽기 전용도, 타임아웃도 — 아무것도 적용되지 않음. getCurrentTransactionName()은 외부 메서드의 이름을 그대로 반환

04 — Experiments

실험 결과

GET

실험 1-A: 자기 호출

같은 클래스 내에서 REQUIRES_NEW 메서드를 직접 호출. this.innerMethodWithRequiresNew()는 프록시를 우회하여 같은 트랜잭션에서 실행됩니다.

outer_tx = "...experimentSelfInvocation" inner_tx = "...experimentSelfInvocation" same_transaction = true // REQUIRES_NEW 무시됨!
GET

실험 1-B: 외부 호출

별도 Bean의 REQUIRES_NEW 메서드를 호출. 프록시를 통과하므로 TransactionInterceptor가 정상 실행되어 새 트랜잭션이 생성됩니다.

outer_tx = "...experimentExternalCall" inner_tx = "...methodWithRequiresNew" same_transaction = false // 정상 동작!
05 — Solutions

해결 방법

권장

별도 서비스 분리

트랜잭션 경계가 필요한 메서드를 다른 @Service 클래스로 이동

+ SRP 준수, 테스트 용이
- 클래스 수 증가
대안

@Lazy Self-Injection

자기 자신의 프록시를 @Lazy @Autowired로 주입하여 self.method()로 호출

+ 최소한의 코드 변경
- 순환 결합 암시, @Lazy 필수
비권장

ApplicationContext

context.getBean()으로 프록시를 직접 꺼내 호출

+ 어디서든 사용 가능
- Service Locator 안티패턴
고급

AspectJ 모드

바이트코드 위빙으로 프록시 없이 AOP 적용. 자기 호출 문제 완전 해결

+ 완전한 AOP
- 빌드 복잡성 대폭 증가
06 — Code

실험 코드

@Transactional fun experimentSelfInvocation(): Map<String, Any> { val outerTxName = TransactionSynchronizationManager.getCurrentTransactionName() // 자기 호출 - 프록시 우회! val innerResult = innerMethodWithRequiresNew() // this.method() result["same_transaction"] = (outerTxName == innerResult["tx_name"]) // true! } @Transactional(propagation = Propagation.REQUIRES_NEW) fun innerMethodWithRequiresNew(): Map<String, Any?> { // REQUIRES_NEW지만 자기 호출이므로 새 트랜잭션이 생성되지 않음 val txName = TransactionSynchronizationManager.getCurrentTransactionName() return mapOf("tx_name" to txName) }
@Transactional fun experimentExternalCall( externalService: TransactionExperimentExternalService ): Map<String, Any> { val outerTxName = TransactionSynchronizationManager.getCurrentTransactionName() // 외부 서비스 호출 - 프록시를 통함! val innerResult = externalService.methodWithRequiresNew() // proxy.method() result["same_transaction"] = (outerTxName == innerResult["tx_name"]) // false! }
@Service class MyService { @Lazy @Autowired private lateinit var self: MyService // 자기 자신의 프록시 주입 @Transactional fun outerMethod() { self.innerMethod() // 프록시를 통해 호출 → 정상 동작! } @Transactional(propagation = Propagation.REQUIRES_NEW) fun innerMethod() { /* ... */ } } // @Lazy를 사용하면 실제 빈 대신 lazy 프록시가 주입되어 순환 참조를 회피한다. // Spring Boot 3.0+에서는 allow-circular-references가 기본 false이므로 // @Lazy 없이는 시작 실패한다.
07 — Myths

흔한 오해와 실제 동작

오해

"private 메서드만 문제된다"

실제

public 메서드도 자기 호출이면 동일하게 무시됨 (이 실험이 증명)

오해

"같은 패키지면 안 된다"

실제

프록시 우회 여부는 this 참조 사용 여부에 달림. 다른 Bean이면 같은 패키지여도 정상

오해

"@Transactional 없는 메서드에서 호출하면 괜찮다"

실제

외부→내부든, 내부→내부든 this.method()면 항상 프록시 우회

오해

"Spring Boot에서는 해결됐다"

실제

Spring Boot도 동일한 프록시 기반 AOP를 사용. 해결되지 않았음

this.method()는 프록시를 우회한다

트랜잭션 경계가 필요한 메서드는 별도 서비스로 분리하세요. 자기 호출이 불가피하면 @Lazy Self-Injection을 사용하세요. REQUIRES_NEW, readOnly, timeout — 모든 속성이 동일하게 영향받습니다.
